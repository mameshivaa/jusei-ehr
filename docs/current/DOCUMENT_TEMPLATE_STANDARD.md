# Document Template Standard (Current)

最終更新: 2026-02-13
文書オーナー: Product/Engineering
レビュー周期: 四半期
承認者: Tech Lead + Security Officer

## 1. 目的

`docs/current/` を複数ベンダー・複数担当者で継続運用しても品質が崩れないよう、文書の最小構造と記述規約を統一する。

## 2. 適用範囲

- `docs/current/*.md` の Tier A / Tier B 文書
- 新規作成文書および大幅改訂文書

## 3. 必須ヘッダー

Tier A / Tier B 文書の冒頭には以下を必ず記載する。

- 最終更新: `YYYY-MM-DD`
- 文書オーナー: 役割名
- レビュー周期: 月次 / 四半期 / 半期
- 承認者: Tier A は必須、Tier B は任意

## 4. 推奨セクション構成

最低限、次の順序を推奨する。

1. 目的
2. 適用範囲
3. 定義/前提
4. 本文（仕様、手順、基準）
5. 例外/エスカレーション
6. 参照文書
7. 改訂履歴（任意）

## 5. 記述ルール

- 断定的な文章で記述し、曖昧語（適宜、なるべく）を避ける。
- 役割責任は `RACI.md` のロール名に合わせる。
- 手順は時系列で番号付きにする。
- 判定基準は「通過条件/失敗条件」を明示する。

## 6. ファイル命名規約

- 英大文字スネークケース（例: `CHANGE_AND_RELEASE_CONTROL.md`）
- 略語は既存文書と整合を取る（API, RACI, QA など）
- 同義文書の乱立を防ぐため、作成前に `MASTER_INDEX.md` を確認する

## 7. 変更時の必須チェック

- `DOCUMENT_REGISTER.md` の `last_reviewed` と該当行を更新
- 参照リンク切れがないことを確認
- 影響があるUI導線（`/docs`, 設定画面）を更新
- `CHANGE_AND_RELEASE_CONTROL.md` の変更区分に従って承認を取得

## 8. 外部委託時の追加要件

- 委託先固有の運用語を持ち込まない
- ベンダー交代後も読める中立表現を使う
- 監査証跡（チケット番号、PR番号）を変更履歴に残す

## 9. 例外運用

緊急対応で即時反映が困難な場合は、72時間以内に本標準へ適合するよう改訂する。
