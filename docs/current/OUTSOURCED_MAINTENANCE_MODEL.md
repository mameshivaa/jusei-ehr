# Outsourced Maintenance Operating Model

最終更新: 2026-02-13
文書オーナー: Product/Engineering
レビュー周期: 四半期
承認者: Product Owner

## 1. 目的

外部委託先が、V-OSSの保守を安全かつ継続的に実施するための運用モデルを定義する。

## 2. 委託範囲

含む:

- バグ修正
- 軽微機能改善
- 定期保守（依存更新、監視、文書更新）
- バックアップ/復旧テスト運用支援

含まない（都度承認）:

- 認証方式の全面変更
- DBの破壊的変更
- 監査/法令関連の方針変更
- 外部連携先追加（レセコン本番接続含む）

## 3. 体制

必須ロール:

- 発注側 Product Owner
- 発注側 Security Officer
- 受託側 Vendor Lead
- 受託側 Application Engineer
- 受託側 QA Engineer

責任境界は `docs/current/RACI.md` を参照。

## 4. チケット運用

優先度:

- P1: 業務停止/重大障害
- P2: 主要機能劣化
- P3: 軽微不具合
- P4: 改善要望

SLA目安:

- P1: 1時間以内初動、24時間以内に暫定策
- P2: 4時間以内初動、3営業日以内に対処方針
- P3: 1営業日以内初動、次スプリントで処理
- P4: バックログ管理

## 5. セキュリティ要件

- 本番アクセスは最小権限
- 共有アカウント禁止
- 作業ログと変更履歴を保持
- 機密情報はチケットに平文記載しない
- 緊急操作は二重承認

## 6. 成果物

毎月提出:

- 月次運用レポート（障害、変更、未解決リスク）
- バックアップ/復旧テスト結果
- 文書更新履歴
- 次月改善提案

## 7. 品質ゲート

全リリースで以下を満たす:

- `npm run type-check`
- `npm run lint`
- 影響範囲に応じたテスト実施
- ドキュメント更新完了
- ロールバック手順確認

詳細は `docs/current/QUALITY_ASSURANCE_GATES.md` を参照。

## 8. 契約終了/再委託対策

- ノウハウを `docs/current` に集約
- 引継ぎは `VENDOR_HANDOVER_CHECKLIST` に従う
- ベンダー固有運用は最小化し、再委託可能性を担保する
