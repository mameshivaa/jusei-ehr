# Change and Release Control

最終更新: 2026-02-13
文書オーナー: Tech Lead
レビュー周期: 月次
承認者: Product Owner

## 1. 目的

変更の安全性、追跡性、再現性を確保する。

## 2. 変更区分

標準変更:

- 既存仕様内の修正
- 通常フローで審査

重要変更:

- 認証/権限
- DBスキーマ変更
- バックアップ/復旧
- 監査ログ仕様
- 連携機能

緊急変更:

- P1障害対応
- 先行適用後に事後審査

## 3. 標準フロー

1. 起票

- 目的、背景、影響範囲、ロールバック案を記載

2. 設計

- API、DB、運用手順、監査影響を明記

3. 実装

- 小さなコミットで変更
- 破壊的変更は段階移行

4. 検証

- 型、lint、テスト、手動確認
- 変更に応じてバックアップ/復旧検証

5. 文書更新

- `docs/current` を必ず同期更新
- `DOCUMENT_REGISTER` の更新要否を判断

6. 承認とリリース

- 重要変更はPO/TL承認
- セキュリティ影響がある場合SO承認追加

## 4. 必須成果物

- 変更チケット
- 実装PR
- テスト結果
- ロールバック手順
- 文書更新差分

## 5. ロールバック基準

以下のいずれかで即時ロールバックを検討する:

- 業務停止
- データ整合性リスク
- 監査ログ欠損
- 認証不全

## 6. 緊急変更の事後処理

- 24時間以内に暫定報告
- 72時間以内に恒久対応方針と文書更新
- 月次レビューで再発防止を合意
